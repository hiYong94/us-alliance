# test-audit

## 목적

테스트 코드를 5 기준으로 진단해 *신호/노이즈 비를 개선*하는 정리 권장을 제시한다.

## 5 기준

1. **독립성** — 다른 테스트 결과·실행 순서·공유 상태에 의존하지 않는가
2. **깨지기 쉬운 테스트** — mock 호출 순서 · timestamp 정확 비교 · 라이브러리 내부 의존 등
3. **핵심 로직 우선** — 도메인 가치 vs trivial 매핑
4. **G-W-T 가독성** — 복잡 시나리오에 Given-When-Then 라벨 명시 여부
5. **간결성 · 가독성 · 독립성 종합** — 신호/노이즈 비

## 실행 단계

1. **인벤토리**

```bash
find src test \( -name "*.spec.ts" -o -name "*.e2e-spec.ts" \) | sort
```

2. **추출** — 각 파일의 `describe` / `it` 헤더 + 본문 길이 + setup 패턴 파악
3. **분류** — 각 케이스를:
   - ✅ 핵심 (도메인 행위 검증)
   - 🟡 가치 있음 (계약 · 경계 검증)
   - 🟠 trivial (라이브러리 동작 · mirror · 단순 매핑)
   - 🔴 brittle (mock 순서 · 라이브러리 내부 의존)
4. **권장 출력**:
   - 삭제 후보 (🟠 + 🔴 일부) — 사유 1줄씩
   - G-W-T 라벨링 (5 케이스 이내) — 복잡 시나리오에만
   - 추가 (gap) — 정상 path 누락 · 외부 계약 미커버 등

## 분류 기준 — 삭제 후보의 명확한 시그널

- **mirror** — 한 메소드의 정/반 케이스가 같은 코드 경로일 때 한 쪽 삭제 (예: markDone / markFailed)
- **library behavior** — `JsonDB.push` 가 파일에 쓰는지, `async-mutex.runExclusive` 가 반환값 그대로 전달하는지 등은 *우리 코드 가치 없음*
- **trivial default** — `description ?? null`, `필터 미지정 시 전체 반환` 같은 기본값 매핑
- **duplicate path** — 다른 테스트가 이미 같은 코드 경로를 통과
- **delegation** — 1줄 위임 메소드 (`firstRun → tick`)

## 분류 기준 — G-W-T 라벨링 우선 케이스

- 비동기 mock 시퀀스가 비자명 (예: Promise.allSettled 안 random 소비 순서)
- hang 트릭 등 setup 자체가 비자명
- 둘 이상의 시나리오 분기를 한 테스트에서 검증

## 사용자 선호 강제

- 많은 수보다 적은 수 + 높은 가치
- 라이브러리 신뢰 — 라이브러리 동작 검증은 우리 책임 아님
- 정상 path 가 빠지지 않게 (e2e 에서 특히 중요)
- 핵심 시나리오에만 G-W-T 라벨링 (남용 = 노이즈)
